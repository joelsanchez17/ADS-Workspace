# Limitation Decision Record

## Purpose and decision boundary

This document records the decision phase preceding further application changes and preparation of the supervisor-facing LaTeX report. It evaluates the current implementation, the existing documentation and assets, and the test evidence already collected. No limitation described here has been fixed as part of this phase, and no processor semantics have been changed.

The authoritative project repository supplied for the final documentation is:

```text
git@github.com:RPTU-EIS/RISCV-pipeline-vizualizer.git
```

At the start of this inspection, the working tree was clean at commit `ff31871` (`first draft`). That commit was already present on the configured `origin/main`; this decision phase did not create or push it. The configured remote in the present parent workspace is not the authoritative project repository above, so the final clone instructions must follow the supplied project identity rather than infer it from this workspace.

## Inspection evidence

The assessment used complete repository searches, direct inspection of the Python, HTML/JavaScript, Scala, setup, validation, and report sources, and small read-only application checks. The most relevant observations are:

- The three publication documents containing clone instructions still use a repository placeholder. No published document or generated asset contains a developer home-directory path.
- `GET /workspace?level=1` returns HTTP 200 with an empty Scala map because the route addresses directories that do not exist.
- A cross-origin preflight from an unrelated origin receives `Access-Control-Allow-Origin: *`. Socket.IO is likewise configured with a wildcard origin.
- The landing page labels Level 3 as “Memory” and Level 4 as “Branches,” although backend Level 3 is the branch/jump course design and Level 4 is the integrated memory/hazard design.
- The `Datapath` checkbox invokes a redraw but its value is never consulted. The adjacent hazard checkbox does affect branch-path opacity.
- Six stopped session directories remain beneath ignored `temp_sessions/`; no validation JVM or application server remains active.
- Process cleanup is implemented for recompilation and server shutdown, but returning to the landing page or disconnecting the browser does not notify the server.
- The VCD test already stops at `coreDone`, with 100 cycles as a hard-coded safety maximum. The interactive test is a separate processor instance and can continue beyond that waveform.
- There is no project test runner or CI workflow. `validate_sessions.py` is a useful validation driver, but it assumes a running server, writes documentation data, leaves session directories, and does not assert architectural results.

## Priority summary

### Recommended low-risk fixes

The most defensible pre-report set is **1, 2, 3, 5–12, 17, 21, 24–26, 28–29, 34–35, and 39–41**. These changes correct project identity, misleading UI text, dead entry points, local-default security configuration, waveform configurability, decoder presentation, and repeatable baseline tests. Most are isolated presentation, configuration, or test changes; none requires processor semantic modification.

### Optional medium-effort fixes

IDs **13, 16, 19–20, 23, 31, and 42–44** are valuable but need more design or test work. Vendoring Monaco is materially larger than copying two minified libraries. Retention and concurrency policies also require an explicit operating model. Browser and waveform automation would improve confidence but are not prerequisites for correcting the report.

### Future architectural work

IDs **14–15, 18, 22, 27, 32–33, and 45** concern legacy-tool cleanup, complete offline deployment, resource architecture, production security, equivalence between two simulations, capability modelling, and CI policy. These should be described proportionately as future development rather than presented as small defects.

### Processor changes requiring supervisor approval

IDs **36–38** concern signed RISC-V semantics in the integrated core and the tests required before changing them. They should not be implemented as documentation cleanup. The current source strongly indicates incorrect signed `SRA` and `SLT` behavior, but dedicated positive and negative operand tests must establish the intended course/reference behavior first.
